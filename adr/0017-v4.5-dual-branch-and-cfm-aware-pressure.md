# ADR 0017 — V4.5 dual-branch detection + CFM-aware pressure attribution

**Status:** Accepted
**Date:** 2026-05-07
**Related:** ADR-0014 (V4 scope), ADR-0015 (outline-based duct detection), ADR-0016 (SMACNA pressure class)

---

## Context

V4 ([`../SOLUTION-DESIGN-V4.md`](../SOLUTION-DESIGN-V4.md)) detected ducts as *outlines*: a single filter chain (rectangle → squarish → ink density → aspect) over the post–`filter_oversized` contour list. Air terminals were a separate stage in `app/cv/terminals.py` driven by template matching, and the segment graph + flow trace + pressure derivation lived downstream of the network builder.

Three things broke in practice:

1. **Terminal recall via template matching was poor on `testset2.pdf`** — ~178 candidates, very few attached to segments because cross-cut-bar detection (the V4 segment-boundary primitive) under-recalled (V4 limitation #2 in the product README).
2. **The full V4 graph build (`build_network`) requires every connector classified.** Without bends, tees, and equipment classified, the graph is incomplete and the flow trace short-circuits. That work is sized at 4–6 weeks.
3. **The user wants to demo end-to-end *now*.** A working pressure-class output on `testset2.pdf` is the deliverable, not a fully simulated network static-pressure budget.

We need duct-and-terminal detection that does not depend on cross-cut-bar recall, and a pressure number per duct that does not depend on the graph being complete.

## Decision

V4.5 keeps V4's contour detection primitive (`find_all_rectangles` + `filter_oversized`) and reorganises everything after it.

**Pipeline (post-`filter_oversized`):**

```
post-oversized contours
  ├─ Duct branch (algorithmic)
  │    filter_is_rectangle → squarish → ink-density → aspect
  │    → duct-grammar VLM ladder (Tesseract@600 → VLM@600/900/1200, image-hash cached)
  │    → median px-per-inch scale → length_ft per duct
  │
  └─ Terminal branch (algorithmic + agent OCR)
       filter_is_circle (4πA/P²) → filter_has_horizontal_divider
       → 3-digit OCR ladder (same Tesseract→VLM ladder, same cache)
       → CFM per terminal

       merge: a contour kept by either branch survives
       ↓
  CFM-aware pressure attribution per duct:
    direct adjacency (≤6 px bbox edge) → terminal CFM exact
    fallback                            → inverse-distance-weighted CFM proxy
                                           across a scale-derived 4 ft neighborhood
                                           (capped at 4000 fpm)
       ↓
  Velocity → Darcy ΔP = f·(L/Dh)·(V/4005)² → SMACNA class
```

The two branches are independent — they consume the same input list and merge results — so disabling one (via the debug panel toggles) cleanly ablates that side.

Implementation:

- Filters: `app/cv/rect_filters.py::filter_is_circle` (`4π·A/P²`), `filter_has_horizontal_divider` (centre-row ink density inside the inner span).
- OCR ladders: `app/pipeline/runner_v4.py::_apply_three_digit_filter`, `_ocr_per_crop_with_retry` (existing, now used for ducts under V4.5). Image-hash cache lives in the same module (`_OCR_TEXT_CACHE`).
- Length: `_populate_duct_lengths` — median of per-duct `pixel_short / first_dim_inches`.
- Pressure: `_attribute_cfm_and_pressure` — direct-adjacency-then-neighborhood-proxy, with `OperationalVars` driving `friction_factor` and SMACNA thresholds.
- Frontend: `V4Progress` (7-stage view with per-bbox sub-status), `V4ResultBar` (inline stats + opacity slider + class shading toggle), `V4OcrOverlay` (selected duct → linked terminal highlight, optional class colouring).

V4's `terminals.py` template-match path and the `build_network` graph stage are **parked**, not deleted. The flow-trace + true-network pressure pipeline they fed is the §5.1 work in the product README; V4.5's direct adjacency is the explicit MVP substitute.

## Why this is the right call

1. **Terminal detection is now a CV pipeline, not template matching.** A bisected circle's signature (high circularity + horizontal mid-row ink) is easier to discriminate than a template-match for the terminal symbol — and it generalises across drawings whose air-terminal glyphs differ.
2. **OCR ladder gates both branches by grammar.** A circle without a 3-digit token is dropped (probably a keynote or grid bubble); a rectangle without a `WxH` token is dropped (probably a callout box). Branches converge on "we read a sensible label" before they vote anything into the result.
3. **Pressure is honest under uncertainty.** `pressure_estimated=true` on the payload distinguishes a measured number (terminal touched the duct, exact CFM) from a heuristic (CFM proxied from neighborhood, or no terminal nearby — fall back to 1500 fpm). The frontend shows "est." next to estimated values; the stat strip counts them. The user knows what's load-bearing.
4. **Caching makes iteration cheap.** Slider-tweak re-runs hit the OCR cache and complete in <1 s. The image-hash key (`mode|size|raw bytes`) is stable across slider tweaks that don't touch the masked image, so the system feels responsive without a queue or worker tier.
5. **Direct adjacency is the right MVP scope.** Full network CFM = sum-over-downstream-terminals = a connected-component graph traversal at minimum (§5.1 first item, ~30 LOC) and a real fluid-network simulation at maximum (months). The MVP delivers a *useful* pressure number on every directly-connected duct without taking on either.

## Consequences

**Positive:**

- `testset2.pdf` end-to-end: 4 ducts measured, 17 terminals read, ~6275 total CFM, length + pressure-class on every duct, ~25–60 s wall clock first run, <1 s on cache-warm re-runs.
- The dual-branch shape composes with future work: connected-component CFM (§5.1) replaces the proxy without touching detection. Connector classification slots in between branches and merge.
- Pressure class is colour-coded on the canvas; user gets a visual heat map without reading numbers.
- The OCR cache is the single biggest lever for demo-time responsiveness.

**Negative:**

- Two filter chains plus two OCR ladders means the runner is longer (~700 LOC). Worth it because the chains are conceptually clean and individually toggle-able for debugging.
- Direct adjacency mis-attributes CFM whenever a duct's "right" terminal is one branch-tap away (~4 ft) and another is *touching* — the touching terminal wins regardless of which corridor it belongs to. Consequence is bounded because both terminals are typically on the same trunk; the proxy fallback handles the rest.
- The neighborhood radius is empirical (4 ft). At extreme drawing scales it might over- or under-include; documented in [`../implementation/README.md`](../implementation/README.md) §6.1, surfaced in backend logs as `v4: CFM-proxy radius = …`.
- V4's `terminals.py`, `build_network`, `flow_trace`, `pressure.compute_pressure` are unused on the live path. Risk: bit-rot. Mitigation: ADR captures the parking; §5.1 of the product README captures the revival plan; tests still cover the math layer.

**Neutral:**

- ADR-0015 (outline-based duct detection) is unchanged — V4.5 still detects contours via that path.
- ADR-0016 (SMACNA pressure class) still drives the class boundaries; only the *velocity input* to the class derivation changed (now CFM-aware per-duct rather than fan-derived per-segment).

## Alternatives considered

- **Finish the V4 network graph first.** Rejected for MVP. Connector classification (bends, tees, transitions, equipment) is sized at 4–6 weeks. The dual-branch + direct adjacency shape ships in days and demos the same UX surface (length, CFM, pressure, class).
- **Use V4's `terminals.py` template matcher and skip the circle filter.** Rejected. Template matching tuned for `testset2.pdf` glyph variants over-fits; the circularity + divider check generalises across drawings without per-drawing template authoring.
- **Skip duct-grammar OCR; derive cross-section purely from rectangle pixel-short side.** Considered. Closer to the V4 A9 fallback. Rejected as the *primary* path because without an OCR-anchored scale the absolute lengths drift across drawings (every drawing gets calibrated against itself, not a known scale). A9 should remain the *fallback* for unlabeled ducts (§5.1 fifth row).
- **Run the network-traversal CFM (connected components) in V4.5.** Tempting — it's ~30 LOC and conceptually clean. Punted to §5.1 because the user explicitly scoped V4.5 as direct-adjacency only ("MVP, days not weeks"). The shape composes additively, so adding it later doesn't break anything.
- **Use a fixed pixel adjacency radius (`80 px`) instead of a scale-derived one.** Was the first cut. Replaced because it depended on `rect_dpi=100` and would silently drift if DPI changed. The scale-derived 4 ft radius is correct across DPIs.

## What this means for existing ADRs

- **ADR-0014** (V4 scope) — assumptions A1–A15 still apply; A6 (cross-cut bars define segments) and A7 (dashed crossings) are explicitly *not* used by V4.5 because the network graph isn't built. They re-engage in §5.1.
- **ADR-0015** (outline-based duct detection) — unchanged. V4.5's duct branch still consumes its contours.
- **ADR-0016** (SMACNA pressure class) — unchanged on the class boundaries; the *velocity input* now comes from the CFM-aware attribution rather than the fan-derived flow trace.
